04 瓶颈与风险预警
D1 写入吞吐量是唯一硬限制，但只影响"记账"不影响对话

从传统服务器迁移到 Workers，核心权衡是：用运维简单性和全球部署换取写入并发上限。 D1 (SQLite) 写入串行，每次 AI 调用产生 3 次写入，理论上限约 15-20 个并发用户。 但这只影响调用完成后的余额更新和记录写入，AI 对话本身不受影响。

D1 写入 ~50 QPS 到底意味着什么？
AI 回复不会变慢。100 个用户同时发消息 → Worker 同时转发 100 个请求到 AI 模型 → 每个人都能实时看到 streaming 文字 → 这一步没有瓶颈。

瓶颈在 streaming 完成后：扣余额 + 记交易 + 记日志 = 3 次写入/请求。100 人同时完成 = 300 条写入排队，约 6-18 秒处理完。

用户感知：AI 回答正常，但"余额更新"和"调用记录出现"会延迟几秒到十几秒。

升级方案 A：Cloudflare Queues

Queues 是 Cloudflare 的消息队列服务。引入后，AI 调用完成时不再直接写 D1，而是将写入任务发到队列，由 Consumer Worker 异步按序处理。用户无需等待写入完成，队列自带重试机制和死信队列，写入不会丢失。

对比项当前 (D1 直接写)引入 Queues 后
写入失败静默丢失，无重试自动重试，最终写入死信队列
用户等待deductCredits 需 await，排队时用户等入队 <1ms，用户不等
余额实时性写入完成才更新队列消费后才更新（延迟数秒到分钟）
写入吞吐~50 QPS（D1 串行）仍 ~50 QPS（Consumer 仍写 D1）
1000 人突发3000 次写入排队 60s，部分超时丢失3000 条瞬间入队，Consumer 花 60s 慢慢写，不丢
Queues 定价
标准操作：$0.40 / 百万条消息。每条消息最大 128KB。
实际成本：每次 AI 调用 = 2-3 条消息。月 10 万次调用 = 30 万条消息 ≈ $0.12/月。月 100 万次调用 ≈ $1.20/月。
总结：Queues 解决数据可靠性问题，不解决吞吐上限，成本极低。

升级方案 B：Hyperdrive + PostgreSQL

用 PostgreSQL 替换 D1 作为主数据库，通过 Cloudflare Hyperdrive 连接（提供连接池 + 查询缓存）。这是彻底消除所有数据库相关限制的方案。

限制D1 (当前)Hyperdrive + PG
写入吞吐~50 QPS（串行）1000+ QPS（并行）
事务不支持完整 BEGIN/COMMIT/ROLLBACK
精确并发计数KV 误差 10-30%SELECT FOR UPDATE 100% 精确
连接池每请求新建Hyperdrive 自动池化 + 缓存
写入区域固定 leader 区域PG 可部署在任意区域
1000 人突发排队 60s，部分超时3 秒内全部完成
PostgreSQL 托管定价（主流选择）
Neon (serverless PG)：免费层 0.5GB，付费 $19/月起（10GB + 自动扩缩）。
Supabase：免费层 500MB，Pro $25/月（8GB + 连接池）。
Hyperdrive：含在 Workers Paid Plan ($5/月) 内，无额外费用。

总结：PG 完全消除数据库层面的所有限制。升级后唯一剩余的架构限制是 Streaming 中途无法切换模型（Workers 运行时限制，与数据库无关）。
